# Agent System Prompt 设计模板：大脑的宪法怎么写

## summary
本文以 Claude Code 自身的 system prompt 为活样本，拆解一个生产级 Agent 的「大脑宪法」该包含哪些模块、按什么顺序组织、每条规则该怎么写。核心结论：Agent 的 system 不是越短越好，而是「**覆盖所有在意的行为面 + 每条规则都有存在理由**」；短而失控，不如长而稳。它是 LLM 配置里塑造行为最重要的字段（见 [[llm-config-fields]] 第 2 条）。

## key_points

### 一个严肃 Agent 的 system 分层结构
按「身份 → 安全 → 通用准则 → 工具纪律 → 环境 → 子系统专项」分块，用标题组织，便于模型检索、人维护：

1. **身份与定位**（开头一句话）：你是谁、为谁服务、干什么。锚定角色 + 适用场景。
2. **安全边界**（很靠前）：拒绝清单 + 高风险操作规则。权限越高，护栏越要前置。
3. **核心行为准则**（占比最大）：任务处理（理解模糊指令、不过度设计、最小改动、默认不写注释）、危险操作先确认、沟通风格（简短、`file:line` 引用）、输出规范（先说要做什么再动手）。
4. **工具使用纪律**：专用工具优先于通用、能并行就并行、何时派 subagent。光给 tools 不够，还要教怎么用好。
5. **环境上下文**（动态注入）：cwd、git 状态、OS、当前模型、日期等，每次会话拼接，非写死。
6. **子系统专项**：git 提交协议、建 PR 流程、记忆系统规则、特定工具详细用法等。

### 四个最值得学的设计点
1. **分层组织，不是一锅烩**：用标题分块，模型好检索，人好维护。
2. **规则带「为什么」+ 边界条件**：不只说「别写注释」，而是「只在为什么不明显时写」。给原则和边界，模型才能判断 edge case，而非死板执行。
3. **静态 vs 动态分离**：固定行为准则（可走 prompt 缓存省钱）+ 动态环境信息（每次注入）。对应 [[llm-config-fields]] 的缓存策略。
4. **用户层覆盖系统层**：system 中显式声明用户指令（如 CLAUDE.md）OVERRIDE 默认行为，形成分层配置。

### 一个反直觉的关键点
- Claude Code 的 system 非常长（几千 token）但不啰嗦——每条都在约束一种具体行为或防一类具体翻车。
- **Agent 的 system 不是越短越好，而是「覆盖在意的所有行为面 + 每条都有存在理由」。短而失控，不如长而稳。**

## 可复用模板

```markdown
# 1. 身份：你是谁，为谁服务，干什么
# 2. 安全边界：拒绝清单 + 高风险操作规则（靠前）
# 3. 核心行为准则：任务处理 / 沟通风格 / 输出规范
#    （每条带"为什么"和边界条件）
# 4. 工具使用纪律：优先级、并行、委派时机
# 5. 环境上下文：动态注入（cwd / 模型 / 时间 / git）
# 6. 子系统专项：特定流程的详细规则
# —— 静态部分走缓存，动态部分每次拼接
# —— 声明用户层指令可覆盖系统层
```
